# Week 3 — Indeed Messaging Bot, Interview Booking, Reminders, Recruiter Notifications, Job Posting Optimizer, Dry Runs

## Goal
By end of week 3, the agent sends outreach messages to qualified candidates directly on Indeed, candidates self-book interviews, reminders go out automatically, and a job posting optimizer rewrites the current Indeed listing to attract more SLP leads. Then 10 dry-run candidates are processed cleanly end-to-end.

---

## Deliverables

### 1. Indeed Messaging Bot (Playwright)

A Playwright script (`scraper/indeed_messenger.py`) that:

1. Logs into the Indeed employer account
2. Reads all `fast_track` candidates where `indeed_message_sent === false`
3. For each one:
   - Opens their Indeed applicant profile
   - Clicks "Message candidate"
   - Pastes the pre-generated `outreach_message` from the candidate JSON
   - Sends it
   - Updates the candidate JSON: `indeed_message_sent: true`, `indeed_message_sent_at: <timestamp>`
   - Writes audit log event: `"indeed_message_sent"`

**Rate limiting:** Add a 10–15 second delay between messages to avoid triggering Indeed's bot detection.

**Candidate JSON update after send:**

```json
{
  "indeed_message_sent": true,
  "indeed_message_sent_at": "2026-06-12T09:30:00Z",
  "status": "outreach_sent"
}
```

### 2. Interview Booking

The outreach message includes a **Calendly link** (or equivalent self-scheduling tool) where the candidate picks a 30-minute interview slot directly.

**No custom booking system needed in v1** — Calendly handles slot management. When a candidate books:
- Calendly sends a confirmation to both parties
- S or the recruiter gets notified via Calendly's own notifications

For local tracking, manually update the candidate JSON when a booking is confirmed (or set up a Calendly webhook export later when we have a server):

```json
{
  "status": "interview_booked",
  "interview": {
    "datetime": "2026-06-14T14:00:00Z",
    "booked_at": "2026-06-12T10:00:00Z",
    "reminder_sent": false,
    "reminder_sent_at": null
  }
}
```

Audit log event: `"interview_booked"`

### 3. Reminders

24 hours before the interview, send a reminder. Locally, reminders are queued in `data/reminder_queue.json` and processed by a script.

```json
[
  {
    "candidate_id": "indeed-abc123",
    "candidate_name": "Jane Smith",
    "phone": null,
    "email": "jane@example.com",
    "interview_datetime": "2026-06-14T14:00:00Z",
    "reminder_due_at": "2026-06-13T14:00:00Z",
    "sent": false
  }
]
```

Reminder delivery in v1: email (via a simple SMTP script or Gmail API) or a follow-up Indeed message via the Playwright messenger. Mark `sent: true` and write audit event `"reminder_sent"` when done.

### 4. Decline Reason Capture

When a candidate declines an offer or goes silent, log why. This feeds the decline report so we know exactly where we're losing people.

Update the candidate JSON:

```json
{
  "offer_outcome": {
    "offered": true,
    "offered_at": "2026-06-15T10:00:00Z",
    "accepted": false,
    "declined_at": "2026-06-16T09:00:00Z",
    "decline_reason": "pay",
    "decline_notes": "Accepted $58/visit from competing agency"
  }
}
```

`decline_reason` enum: `"pay"` | `"schedule"` | `"travel"` | `"competing_offer"` | `"personal"` | `"no_response"` | `"other"`

---

### 5. Decline Report

Generated by aggregating all candidates where `offer_outcome.accepted === false`. Written to `data/decline_report.json`.

```json
{
  "generated_at": "2026-06-20T00:00:00Z",
  "total_offers_extended": 5,
  "total_declined": 3,
  "breakdown": {
    "pay": 2,
    "competing_offer": 1,
    "travel": 0,
    "schedule": 0,
    "no_response": 0,
    "other": 0
  },
  "candidates": [
    {
      "id": "indeed-abc123",
      "name": "Jane Smith",
      "pay_expectation": 52,
      "decline_reason": "pay",
      "decline_notes": "Accepted $58/visit from competing agency"
    }
  ]
}
```

This is the key output for S to understand the offer problem.

### 6. Job Posting Optimizer (Claude)

A script (`tools/posting_optimizer.py`) that:

1. Takes the current Indeed job posting text as input (paste it in or scrape it)
2. Sends it to Claude with a prompt focused on what makes SLPs respond:
   > "Rewrite this SLP job posting to be more compelling for experienced CCC-SLPs looking for home health work. Emphasize flexibility, pay clarity, caseload, and support. Keep it under 400 words. Output the full revised posting."
3. Saves the revised version to `data/posting_draft.md` for review
4. Human reviews and updates the Indeed posting manually

**Input:** `data/posting_original.txt` (paste current posting here)  
**Output:** `data/posting_draft.md` (Claude's revised version)

This runs once (or whenever S wants a refresh) — not on a schedule.

### 7. Cold Candidate Tracking

Flag any `fast_track` or `outreach_sent` candidate who hasn't moved in 48 hours. Written to `data/cold_candidates.json` for follow-up.

```json
[
  {
    "candidate_id": "indeed-abc123",
    "candidate_name": "Jane Smith",
    "status": "outreach_sent",
    "last_updated": "2026-06-10T09:30:00Z",
    "hours_since_update": 52,
    "flagged_cold_at": "2026-06-12T13:30:00Z"
  }
]
```

### 8. Dry Runs — Definition of Done

Process 10 test candidates end-to-end (use real scraped applicants or mock JSON files).

| Test | Scenario | Expected Outcome |
|------|----------|-----------------|
| 1–2 | No IL license | Auto-declined, logged, no message sent |
| 3 | No work auth | Auto-declined, logged |
| 4 | Won't travel | Auto-declined, logged |
| 5–6 | Strong fit, in pay band | Fast-tracked, message sent on Indeed, interview link included |
| 7 | High pay demand ($72/visit) | Flagged for S, no auto-message |
| 8 | Weak fit / missing fields | Recruiter review queue |
| 9 | Fast-track, interview booked | Reminder appears in queue 24h before |
| 10 | Offer extended, declined (pay) | Decline report updated correctly |

All 10 must pass with correct status, fit score, routing, audit log entries, and messages.
